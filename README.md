# Rougelike 隨機地圖生成器

## Proposal Report

### 動機與目標
Rougelike 遊戲對我來說的迷人點在於其過程的不可預測性，以及重複遊玩時每次都有不同體驗。  
本專案目的為設計一款 Rougelike 遊戲的隨機地圖生成器，方便在開發類似遊戲時可以使用，期望能製作一套能穩定生成合理、具挑戰性且不重複之地圖的系統  

### 預期功能
參考遊戲《殺戮尖塔》的地圖設計，採用節點式的地圖，給予玩家策略選擇權。
1. 確保地圖路線可行性，防止生成出不合理或無法使用的地圖。
2. 建立節點屬性的分配規則（如：戰鬥、菁英、事件、商店、營地），並能自訂權重，防止生成極端不平衡的路線（例如連續三個營地）。
3. 控制節點分配與路線連接，確保視覺上清晰易讀，不會產生過於混亂的交叉線條。

### 使用技術
預計使用 Godot 當作主要的開發引擎，因為其內建的節點系統非常好用，可以方便快速打樣與實作，也方便視覺化。  
搭配的程式語言是 GDScript，為 Godot 官方的推薦語言。  
核心演算法主要會用到 DAG 的生成，以及在驗證走法時會用到 DFS/BFS 的走訪。

### 時程規劃
* **W7：專案提案**
    * 確立專案範圍、目標與技術堆疊（Godot + GDScript）。
    * 規劃地圖的基礎參數（如：總樓層數、每層最大/最小節點數）。
* **W8：基礎架構與核心資料結構**
    * 在 Godot 中建立專案，設定好適合 Mac 開發與展示的視窗解析度。
    * 以 GDScript 實作基礎的資料結構：定義 `Node`（節點）與 `Edge`（邊）的 Class。
    * 實作初步的 DAG 生成邏輯，能以純文字或陣列形式印出簡單的分層結構，確保層級間是單向推進的。
* **W9：連線演算法與連通性驗證**
    * 撰寫樓層間的連線演算法，處理路線的「分岔」與「合併」。
    * 實作 DFS 或 BFS 走訪，驗證每一條底層的起始路線，最終都必定能通往最頂層的 Boss 節點。
    * 處理防呆機制，消除沒有連外路徑的「孤島節點」。
* **W10：座標轉換與初步視覺化**
    * 將抽象的圖論資料轉換為畫面上實際的 2D 座標 (X, Y)。
    * 利用 Godot 的 `_draw()` 函數或基礎的 `Control` 節點，將節點圖示與連接線初步繪製到螢幕上。
    * 調整節點間距，確保畫面不會過度擁擠。
* **W11：視覺優化與佈局演算法**
    * 優化連線邏輯：實作交點最小化，盡可能避免路線交叉，提升地圖易讀性。
    * 加入隨機偏移量：讓每個節點的 X 座標產生微幅的隨機偏移，使整體路線看起來更自然，而非死板的棋盤格。
* **W12：節點屬性與動態權重分配**
    * 實作帶權重的隨機選擇演算法，確保在生成節點類型時效能良好。
    * 分配詳細的節點屬性。
    * 加入生成限制規則（例如：連續兩層不能都是營地，或是第一層必定是普通戰鬥）。
* **W13：UI 介面與玩家互動體驗**
    * 將臨時的圖形替換較為美觀美術素材或更精緻的 Godot UI 元件。
    * 實作滑鼠點擊的互動邏輯：加入高亮效果，並確保玩家點擊當前節點後，只能選擇與其相鄰且有連線的「下一層」節點，其餘節點反灰不可選。
* **W14：壓力測試、除錯與參數化**
    * 建立一個簡單的控制面板 UI，讓使用者可以即時調整參數（如：節點密度、各類型機率、生成種子 Seed）並重新生成地圖。
    * 進行大量的隨機生成測試，確保無論參數怎麼調整，演算法都不會當機或產出死路。
    * 準備並錄製不同參數下地圖生成的素材。
* **第十五週：發表與 Demo 影片**
    * 剪輯與後製 Demo 影片，完整展示「參數調整 $\rightarrow$ 地圖生成 $\rightarrow$ 點擊推進路線」的流暢過程。
    * 統整 GDScript 程式碼，完成期末技術文件。
    * 進行期末專案發表與展示。

### 與課程的關聯
1. DAG  
最核心的功能，涉及各種圖論中關於有向無環圖的隨機生成，以及該用何種資料結構去儲存圖
2. 圖的走訪  
在完成生成後，可以透過 DFS/BFS 走訪去驗證地圖的可行性，以及確定節點的深度等
3. 隨機演算法  
實作帶有權重的隨機選擇演算法。
